# Code Modification Behavior Constraints

> 代码修改行为约束。所有 AI 编码助手在此仓库中修改任何文件前，必须遵守本文件列出的原则。
> 与 `AGENTS.md`（外部信息来源 + 代码质量门禁）和 `CLAUDE.md`（Claude Code 工具约定）并列。

**核心精神**：文档不可信，代码是唯一真相；取巧即垃圾；可局部移植才是有用代码；修改者必须对引入的 Bug 负全责。

---

## 1. 修改前验证

- 修改任何文件前，必须**重新阅读**该文件的完整上下文，不得基于记忆或假设推断代码行为。
- 必须确认当前代码的**实际运行逻辑**，禁止以文档、注释、commit message、CHANGELOG 中的描述作为逻辑依据。
- 若修改涉及"已修复"的 Bug，必须先在**代码层面**验证该修复的真实质量，确认非取巧后再决定是否保留或重构。

**反模式**：arch report 写"火山口有 N 个 unwrap"，但 N 是统计脚本把测试代码混进生产代码的产物。**只有重新统计 + 抽样验证**才能发现这条 claim 是分类错误。

---

## 2. 根因修复原则

- 每次修改必须针对**根因**，禁止添加：
  - 临时补丁 / workaround
  - magic numbers
  - 硬编码条件分支（仅在表面"修"了而不解决架构问题）
  - try-catch 吞异常
- 若根因涉及**架构设计**，必须提出架构级修改方案，不得用局部 hack 掩盖。
- 修改后，原取巧代码**必须被删除或重构**，不得保留为"兼容代码"。

**反模式**：v0.53 patch 修了 `eval_binary` Sub/Mul/Div/Mod 但漏了 Add —— 这是"修了 4/5 取巧型修复"。修改完成后必须复检兄弟路径。

---

## 3. 可移植性保护

- 修改不得引入新的平台相关依赖、编译器特定扩展、或隐式环境假设。
- **不得使用全局状态**传递修改意图；若必须共享状态，需通过显式接口参数传递。
- 修改后的模块应可被**直接复制到其他项目中独立运行**，耦合度不得增加。
- **禁止修改构建脚本或配置文件**来"适配"代码逻辑问题。

**判断标准**：把修改后的模块切到另一个 Rust 项目里能否直接用？做不到的修改就是违规。

---

## 4. 清晰度提升

- 修改后的代码**必须比修改前更易读**。若为了修复 Bug 导致代码更难理解，说明修复方案错误。
- **命名必须自解释**：禁止用 `flag1`、`tmp`、`fix_bugs_2024` 等无意义命名。
- 复杂逻辑必须拆分为可独立理解的单元，禁止用注释解释"这段代码为什么这么写"。
- **删除因修改而失效的注释**，禁止保留与代码行为不符的注释。

**反模式**：保留 `// v0.53 根因修复：...` 这种实现历史作为注释 —— 实现历史属于 git blame，doc comment 应描述**当前代码行为**。

---

## 5. 修改范围控制

- **最小化** ≠ "只加一行 if 判断"。最小化 = "只改必要的部分"。
- 禁止在修改中**顺手做无关重构**；若发现其他问题，另开任务处理。
- 跨文件修改必须**显式列出所有被修改文件及修改理由**，禁止隐式依赖。

**反模式**：用户说"修这个 Bug"，改完顺手把 5100 行的文件拆了。这是借机重构，违反红线 5。

---

## 6. 回归验证

- 修改完成后，必须验证修改**未破坏原有可移植代码的行为**。
- 必须提供验证步骤：如何证明这个修改没有引入新 Bug，如何证明根因已被消除。
- 若无法本地验证，必须在修改说明中**明确标注风险点**。

**mora-lang 验证清单**：
1. `cargo build --all-targets` 通过
2. `cargo test --all` 通过（库 + 集成 + doc）
3. `cargo clippy --all-targets --all-features -- -D warnings` clean
4. `cargo fmt --check` 零 diff

---

## 7. 交付规范

每次修改必须输出：
1. **修改文件**：精确路径
2. **修改位置**：行号或行号范围
3. **修改前代码**：diff BEFORE
4. **修改后代码**：diff AFTER
5. **根因说明**：为什么会这样、写代码/读 doc 不够，必须回到代码
6. **可移植性影响评估**：修改后的模块是否仍可独立移植

禁止输出"大概改一下"或"参考这个思路"等模糊描述，必须给出可直接应用的精确修改。

若修改涉及删除取巧代码，必须说明：
- 该取巧代码原本试图掩盖的问题
- 现在的根治方案

---

## 【红线】

- ❌ **用文档替代代码分析**
- ❌ **新增 workaround 或临时补丁**
- ❌ **引入平台绑定或全局状态依赖**
- ❌ **修改后代码清晰度下降**
- ❌ **修改未验证或无法说明根因**
- ❌ **保留旧取巧代码作为"兼容"**

---

## 核心断言

> **代码跟垃圾的唯一区别，就是可移植性和清晰度。修改必须让代码离垃圾更远，而不是更近。**

## 与仓库其它约束的关系

| 文件 | 约束范围 |
|------|---------|
| `AGENTS.md` | 外部信息来源（MCP 搜索）+ 代码质量门禁（build/test/clippy/fmt）+ unwrap/panic 政策 |
| `CLAUDE.md` | Claude Code 工具约定（CLAUDE 特有的提示） |
| 本文件 | 代码修改**行为**约束（7 原则 + 红线）—— 适用于任何 AI 助手写代码前 |

本文件不与 `AGENTS.md` §3（代码质量）冲突，而是从**行为层面**对其补充：
- `AGENTS.md` §3 说"测试必须通过、禁止 unwrap"
- 本文件说"修改时如何决策才能既通过测试又不引入取巧"
